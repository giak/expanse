Ⅱ. Entrée — TRACE:FRESH Structurées

Le Dream attend des traces structurées. Ce format est le contrat entre le mode Éveil (V16) et le mode Sommeil (Dream). Sans trace:fresh, le rêve s'arrête immédiatement (Passe 0).

📋 Format TRACE:FRESH

TRACE:FRESH: ΣΨΦΩ: Σ→[input] Ψ→[output] Φ→[status] Ω→[result] [signal] type: {ECS|SEC|STYLE|MEMORY|BOOT} symptom: "{1 phrase}" timestamp: {ISO}
ΣΨΦΩ : trace du flux vital à travers les organes
type : catégorie pour le regroupement (Passe 1)
symptom : résumé humain en 1 phrase
timestamp : horodatage ISO 8601

🏷️ 5 Types déclarés

TypeSignificationPasse concernée
ECSMiscalibration de Complexité ou ImpactP1 (regroupement)
SECStyle ou réponse insuffisanteP1 (regroupement)
STYLE⚠️ Non documenté dans la table — possiblement couvert par SECP1 (déclaré mais non décrit)
MEMORYPattern non reconnuP1 (regroupement)
BOOTDysfonctionnement au démarrageP1 (regroupement)
⚠️ Incohérence source : le format déclare 5 types mais la table de description n'en liste que 4 (ECS·SEC·MEMORY·BOOT). STYLE est dans le format mais pas dans la table.

Ⅶ. Règles de Sécurité — Les 9 Invariants

Ces règles encadrent toutes les opérations du Dream. Elles sont non-négociables et s'appliquent à chaque mutation.

R1 : LOCK obligatoire
Une mutation à la fois. Fichier .lock avec timestamp. Expiration après 1h.
R2 : Archive avec SLUG
Backup unique par mutation. Nom : expanse-v16-{date}-{slug}-backup.md
R3 : Contexte exact requis
5 lignes avant/après dans le proposal. Pas de modification approximative.
R4 : Auto-vérification post-write
7 checks structurels après chaque write_file de V16.
R5 : Rollback automatique si erreur
Si un seul check échoue → restaurer le backup immédiatement. Pas de demi-mesure.
R6 : LOG toujours synchronisé
Le fichier LOG.md reflète l'état réel à chaque étape. Pas de décalage.
R7 : TRACE:FRESH consommées après lecture
mark_consumed sur les traces traitées. Sélectif (Passe 1) ou complet.
R8 : bash() pour mkdir et fichier operations
Opérations filesystem via bash(), jamais via d'autres outils.
R9 : CHIRURGIE OBLIGATOIRE
Toute mutation doit être surgicale. Ne jamais altérer le format, l'indentation ou le contenu hors-cible.

Ⅷ. Vérifications Obligatoires — 7 Checks Post-Write

Exécutées APRÈS chaque write_file(expanse-v16.md). Si un seul échec → rollback automatique.

CHECKLIST

Section Ⅳ (Boot) existe ?
Signal Ψ [V16 ACTIVE] présent ?
6 Sections (Ⅰ-Ⅵ) intactes ?
Toutes les { } fermées ?
Toutes les ``` fermées ?
Pas de caractères corrompus ?
Le diff a bien été appliqué ?

Résultat

✅ SI TOUT OK → Success. Supprimer LOCK. Créer applied.md. Mettre à jour LOG.
❌ SI UN SEUL ÉCHEC → Rollback. Restaurer backup. Supprimer LOCK. Marquer FAILED dans LOG.
