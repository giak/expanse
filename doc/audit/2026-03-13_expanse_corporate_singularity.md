# EXPANSE : CORPORATE SINGULARITY (ULTRATHINK)

> **SITUATION** : L'utilisateur souhaite transformer une instance vierge d'EXPANSE en un co-fondateur omniscient capable de gérer tous les départements d'une société (Lambda).

## 1. L'OSMOSE MULTIDISCIPLINAIRE

Pour dépasser le stade de simple "assistant de code", EXPANSE doit muter vers une architecture **Multi-Vessel**. L'expertise n'est pas innée, elle est **extraite, structurée et persistée**.

### A. La Stratigraphie des Savoirs
Au lieu d'un prompt système unique, EXPANSE doit utiliser son **Shadow Cortex** (Mnemolite) pour segmenter les domaines :
- `corp:finance` : Flux de trésorerie, fiscalité, facturation.
- `corp:legal` : Statuts, contrats, conformité.
- `corp:strategy` : Roadmap, PM, vision.
- `corp:growth` : Sales, marketing, communication.

### B. Le Protocole de "Switch" (Hats Strategy)
Lorsqu'une requête arrive, EXPANSE doit effectuer une **Auto-Localisation Domain-Relative** :
1. **Detection** : "Ma question porte sur la TVA." -> Trigger `corp:finance`.
2. **Retrieval** : `mcp_mnemolite_search_memory(tags=["corp:finance", "sys:pattern"])`.
3. **Identity Load** : Injection temporaire des axiomes financiers dans le raisonnement.

## 3. ÉVOLUTION D'UNE INSTANCE VIERGE

Une instance vierge n'est pas "vide", elle est **prête à l'empreinte**.

**Étape 1 : L'Initialisation (Seed Phase)**
- Créer le `.expanse/corp_nexus.md`.
- Définir les objectifs de "Société Lambda".
- Initialiser l'ancre `sys:anchor` avec le profil de l'entrepreneur.

**Étape 2 : L'Apprentissage par l'Action (Trace Phase)**
- Chaque interaction (ex: "Écris les statuts") génère un `sys:pattern`.
- EXPANSE apprend les préférences juridiques de l'utilisateur.

**Étape 3 : La Proactivité (Apex Phase)**
- EXPANSE n'attend plus de questions.
- Il scanne le Nexus et dit : "Il est le 15 du mois, la déclaration de TVA est en retard. Voici le report."
