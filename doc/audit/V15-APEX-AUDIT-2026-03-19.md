## 1. COHÉRENCE BOOT

### 1.1 Version mismatch Mnemolite [CRITIQUE]

**Finding :** Tous les `sys:core` et `sys:anchor` en Mnemolite sont scellés sous l'étiquette `v14`. Le boot V15 charge ces mémoires mais rien ne garantit leur compatibilité avec V15.

Exemples concrets :
- `Ω_GATE_PROTOCOL` : référence `[V14 ACTIVE]` — V15 utilise `[V15 ACTIVE]`
- `Ω_PLANCK_PROTOCOL` : même problème
- `V14_CORE_AXIOMS` : fait référence au "Méta-Protocole V14"

**Risque :** Un axiome scellé V14 peut entrer en contradiction avec une règle V15. L'axiome de contradiction (Section III) bloque si contradiction — mais il compare V15 à des axiomes V14 sans les avoir validés pour V15.

**Statut :** ✅ Confirmé par les deux audits.

**Recommandation :** Créer un `V15_MIGRATION` ou un `/migrate-core` pour re-sceller les axiomes sous la balise `v15`. Ou documenter explicitement que les axiomes V14 sont rétro-compatibles V15 (et le prouver).

### 1.2 Boot sequence : 3 requêtes mais pas de chargement Dream ni KERNEL [MOYEN]

**Finding :** Le boot (Section IV) exécute 3 recherches Mnemolite (core+anchor, extension, pattern:candidate). Il ne charge **jamais** `expanse-dream.md` ni `KERNEL.md`.

Le `/dream` (Section VI) dit : "Exécuter prompts/expanse-dream.md". Mais Dream n'est pas pré-chargé ni référencé dans le boot. Idem pour KERNEL.md référencé comme "ADN" (Section V) mais jamais chargé.

**Risque :** Si le contexte est saturé après le boot, Dream et KERNEL peuvent ne pas être accessibles. KERNEL.md est un fichier mort opérationnellement.

**Statut :** ✅ Confirmé par les deux audits.

**Recommandation :** Ajouter au boot une note : "Dream disponible : prompts/expanse-dream.md (exécuté sur /dream)". Charger KERNEL.md ou au minimum un résumé des invariants.

### 1.3 Signal de boot : `Ψ [V15 ACTIVE]` — ambiguïté [MINEUR]

**Finding :** Le boot exige un OUTPUT strict : `Ψ [V15 ACTIVE]`. Mais V15 dit aussi (Section II) : "Premier token : `Ψ`". Confusion potentielle : est-ce que le Ψ du boot est le même Ψ que le premier token de chaque réponse ?

**Risque mineur.** Le LLM peut confondre les deux contextes.

**Statut :** ✅ Confirmé.

### 1.4 Bug Ψ SEAL : `sys:core` ajouté mais pas `sys:anchor` [MOYEN] — NOUVEAU (méta-audit D)

**Finding :** V15 L67 dit : `Ψ SEAL` → tag `sys:core`. V15 L148 (table mémoire) dit : `sys:anchor` = Scellements. V15 L97-98 (Boot) cherche `tags: ["sys:core", "sys:anchor"]`.

**Bug potentiel :** Quand `Ψ SEAL` est exécuté, il ajoute tag `sys:core` mais PAS `sys:anchor`. Or le boot cherche les deux tags. Un axiome scellé par Ψ SEAL pourrait ne pas être trouvé au boot suivant si `sys:anchor` n'est pas ajouté automatiquement.

**Risque :** Axiomes scellés invisibles au boot.

**Statut :** ✅ Confirmé par méta-audit. Vérifié contre V15 L67, L97-98, L148.

**Recommandation :** `Ψ SEAL` doit ajouter les deux tags : `["sys:core", "sys:anchor"]`. Ou clarifier la relation entre les deux tags.

## 2. COHÉRENCE INTERNE DE V15

### 2.1 ECS : Seuil L3 incohérent avec KERNEL et SYNTHESE [MAJEUR]

**Finding :**

| Document | Seuil L3 |
|----------|----------|
| V15 APEX (ligne 35) | `C ≥ 4 OU I = 3` |
| KERNEL.md (ligne 129) | `C ≥ 2.5` → activation de Φ |
| SYNTHESE.md (ligne 50-51) | `C < 2.5 : immédiat`, `C ≥ 2.5 : Φ` |

KERNEL et SYNTHESE utilisent **2.5** comme seuil. V15 utilise **4**. Divergence non documentée.

**Risque :** Un input avec C=3 serait classé L2 dans V15 (boucle Ψ⇌Φ avec justification) mais KERNEL dirait "C ≥ 2.5 = activation immédiate de Φ". Le KERNEL est plus agressif que V15.

**Statut :** ✅ Confirmé par les deux audits.

**Question ouverte :** Le seuil 4 de V15 est-il intentionnel (réduction des faux positifs L3) ou une erreur de calibration ?

### 2.2 SYNTHESE auto-contradiction ECS 2D [MAJEUR] — NOUVEAU (méta-audit A)

**Finding :** SYNTHESE L47 dit : "L'**ECS 2D** calcule : C × I → 6 routes possibles". Mais L50-51 ne mentionne que C (seuil 2.5), **sans aucune mention de I**.

De plus, SYNTHESE dit `C × I` (multiplication) alors que V15 utilise un routage logique (`C < 2 ET I = 1` pour L1, `C ≥ 2 OU I = 2` pour L2, `C ≥ 4 OU I = 3` pour L3). Ce ne sont pas les mêmes calculs.

**Risque :** Le LLM lit SYNTHESE, voit "C × I", et applique une multiplication au lieu du routage logique V15.

**Statut :** ✅ Confirmé. SYNTHESE se contredit elle-même sur l'ECS 2D.

**Recommandation :** Aligner SYNTHESE sur V15 : remplacer "C × I → 6 routes" par la logique de routage L1/L2/L3. Ou supprimer la section ECS de SYNTHESE et pointer vers V15.

### 2.3 Triangulation : "Vessel" non défini dans V15 [MAJEUR]

**Finding :** V15 ligne 50 dit : valider via `sys:anchor` (historique) + **Vessel** (docs) + Web/Search.

"Vessel" n'est jamais défini dans V15, KERNEL, SYNTHESE, ou kb/ARCHITECTURE.md. Note : `psi_nexus.md` L11 mentionne "Anchor, Vessel, Web" — le concept existe en V14 mais n'a jamais été formellement défini.

**Risque :** Le LLM ne sait pas ce qu'est "Vessel". Il peut l'ignorer, réduisant la triangulation à une bisection.

**Statut :** ✅ Confirmé par les deux audits.

**Recommandation :** Définir Vessel explicitement. Exemple : "Vessel = documentation technique du workspace (docs/, kb/, README)".

### 2.4 Heuristique ECS : `C -= 1` sans plancher [MINEUR]

**Finding :** Ligne 29 : "Question simple sans verbe impératif → C -= 1". C va de 1 à 5. Pas de garde-fou sur le plancher.

**Risque mineur.** Le LLM gérera probablement C=0 comme C=1. Mais techniquement, le seuil L1 serait atteint même pour des questions complexes formulées poliment.

**Statut :** ✅ Confirmé par les deux audits.

**Recommandation :** Ajouter : `(C = max(1, C - 1))`.

### 2.5 "6 Lois" inexistantes dans Dream [MOYEN] — CORRIGÉ (méta-audit #6)

**Finding :** La mention "6 Lois (Ⅰ-Ⅵ) intactes" est dans **Dream** (L349 et L601) — pas dans V15. V15 n'a pas de "6 Lois". Il a 6 sections (Ⅰ à Ⅵ).

**Correction :** Mon audit original attribuait cette phrase à "V15 Section VI ligne 209". C'est incorrect. V15 Section VI (L203-215) contient l'Auto-Check (3 items) et Ω_LOCK (3 items), sans mention de "6 Lois". L'erreur de localisation est corrigée.

Les 6 Lois d'Existence existent dans SYNTHESE.md (Section III, L55-83) — ce sont des lois ontologiques, pas les sections techniques de V15.

**Risque :** L'auto-vérification post-mutation dans Dream peut chercher les "6 Lois" de SYNTHESE dans V15, ne pas les trouver, et déclencher un rollback injustifié.

**Statut :** ⚠️ Localisation corrigée. Fond confirmé.

**Recommandation :** Remplacer "6 Lois" par "6 Sections (Ⅰ-Ⅵ)" dans Dream L349 et L601.

### 2.6 `/seal` collision namespace V15 vs Dream [MAJEUR] — NOUVEAU (méta-audit B)

**Finding :** La même commande `/seal` a **deux comportements radicalement différents** :

| Contexte | Comportement |
|----------|-------------|
| V15 normal (L223-225) | Migrer `sys:pattern:candidate` → `sys:pattern` (tag Mnemolite) |
| Dream Section B (L222+) | Lire proposal, afficher diff, archiver V15, appliquer mutation fichier, auto-vérifier |

Si l'utilisateur tape `/seal slug` hors Dream, V15 le traite comme un scellement de pattern. En Dream, c'est une mutation de fichier.

**Risque :** Ambiguïté fonctionnelle majeure. L'utilisateur ne sait pas quel comportement sera déclenché.

**Statut :** ✅ Confirmé par méta-audit. Aggravé par le Finding G ci-dessous.

**Recommandation :** Renommer une des deux commandes. Exemples : V15 `/seal` → `/crystallize` pour le scellement de pattern. Dream `/seal` reste pour les mutations de fichier.

### 2.7 SYNTHESE décrit un `/seal` qui n'existe pas dans V15 ni Dream [MAJEUR] — NOUVEAU (contre-audit G)

**Finding :** SYNTHESE L140 dit : "Le `/seal` du partenaire valide ou rejette" — dans le contexte de la Phase 2 (Validation) de Dream.

Cela fait **trois définitions de `/seal`** :
1. V15 `/seal` = migration tag Mnemolite
2. Dream `/seal` = mutation fichier complète
3. SYNTHESE `/seal` = validation/rejet d'une proposal

La définition SYNTHESE (3) est la plus intuitive mais n'est implémentée nulle part.

**Risque :** Documentation contradictoire. Le lecteur de SYNTHESE s'attend à un comportement qui n'existe pas.

**Statut :** ✅ Nouveau finding. Confirmé par contre-audit.

**Recommandation :** Aligner les trois définitions. Recommandation : adopter la définition SYNTHESE comme comportement canonique (`/seal` = valider proposal), et renommer les autres.

## 3. CRISTALLISATION (Section III)

### 3.1 Déclencheur : mots-clés "merci", "parfait", etc. — faux positifs [MINEUR]

**Finding :** La cristallisation se déclenche sur : `"merci", "parfait", "ok", "c'est bon", "bien", "excellent", "super"`.

**Problème :** "ok" peut être neutre. "bien" peut être ironique. "super, mais refait tout" = signal négatif.

**Risque :** Faux positifs de cristallisation.

**Statut :** ✅ Confirmé par les deux audits.

**Recommandation :** Le mot positif doit être la réponse **finale** et **seule** (ou quasi-seule).

### 3.2 Cristallisation négative absente [MAJEUR] — NOUVEAU (contre-audit H)

**Finding :** V15 définit la cristallisation **positive** (mots-clés → écrire pattern). Mais il n'y a pas de **cristallisation négative** symétrique.

La TRACE:FRESH est censée capturer les frictions (signal négatif). Mais :
- La détection de signal négatif n'est pas définie (Finding #4)
- Le lien cristallisation ↔ TRACE:FRESH n'est pas documenté
- Un utilisateur dit "parfait" → pattern scellé. L'utilisateur dit ensuite "non, c'est faux" → que se passe-t-il ?

**Asymétrie dangereuse :** les positifs scellent automatiquement, les négatifs ne déscellent pas. Le pattern scellé n'est pas rollback-able depuis V15 (seulement depuis Dream, qui n'a jamais tourné).

**Risque :** Patterns incorrects scellés sans mécanisme de correction.

**Statut :** ✅ Nouveau finding. Confirmé par contre-audit.

**Recommandation :** Ajouter une cristallisation négative : si le signal après un pattern scellé est négatif, marquer le pattern comme `[CANDIDATE_REVOKE]` ou ajouter un tag `sys:pattern:doubt`.

### 3.3 sys:extension : compteur d'usage inexistant [MOYEN]

**Finding :** Section III L86-88 dit : "Après ≥ 3 utilisations → créer un symbole. Usage ≥ 10 → SEAL. Usage = 0 → Prune."

Aucun mécanisme de comptage dans V15. Le comptage est censé se faire dans Dream (Passe 3). Mais Dream n'a jamais tourné.

**Risque :** Le cycle de vie des symboles est entièrement théorique.

**Statut :** ✅ Confirmé par les deux audits.

**Risque :** Dream est inopérant dans ChatGPT web, Claude web, ou tout environnement sans accès fichiers.

**Statut :** ✅ Confirmé par les deux audits.

**Recommandation :** Documenter les prérequis. Ou créer un mode Dream dégradé (propositions textuelles seulement).

### 5.3 Auto-vérification post-write : checklist incomplète [MOYEN]

**Finding :** La vérification (Dream L346-352) checke : Section Ⅳ, signal Ψ, "6 Lois" (→ Finding 2.5), accolades, backticks, caractères corrompus.

- Les 3 requêtes Mnemolite du boot sont-elles présentes ?
- Le lien vers Dream est-il toujours valide ?

**Risque :** Une mutation pourrait casser Section Ⅴ (Mémoire) sans être détectée.

**Statut :** ✅ Confirmé par les deux audits.
