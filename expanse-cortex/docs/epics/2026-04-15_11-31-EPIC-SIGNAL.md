## 🎯 PROBLÈME

Les 7 vues actuelles du Cortex montrent la **structure** (ce qui EST), pas le **processus** (ce qui SE PASSE). C'est comme avoir un plan d'un moteur sans jamais le voir tourner. La vue Pipeline est la plus utile car elle montre la direction du flow Σ→Ψ→Φ→Ω→Μ — mais elle reste statique.

**La question centrale** : *Que se passe-t-il quand un LLM charge Expanse ?* On veut le VOIR.

---

## 💡 VISION

> **Le Signal est un simulateur. Pas un miroir.**

Les autres vues reflètent l'état du cortex (nœuds, arêtes, métriques). Le Signal **simule le processus cognitif** : un signal (input utilisateur) traverse les organes en temps réel, avec chaque étape du processus visible et animée.

Ce que fait Expanse dans la tête d'un LLM — rendu visible, pas-à-pas, avec des données réelles ou fake.

## 🎬 SCÉNARIOS

Les scénarios sont le cœur du Signal. Chaque scénario modélise un processus cognitif distinct d'Expanse V16.

### 1. L1 — Trace fulgurante

**Quand** : Question simple, C<2 ET I=1
**Processus** : Σ → Ω direct (1 phrase max). Pas de boucle audit.

```
Step 1: Σ Input reçu → "quelle heure est-il ?"
        ECS: C=1, I=1 → L1
Step 2: Ω Synthèse directe → 1 phrase max
Step 3: Μ Émission → Signal OK, cristallisation légère
```

**Données visuelles** :
- Particule va directement Σ→Ω (saute Ψ et Φ)
- ECS badge vert (L1)
- Aucun tool call
- Durée totale simulée : ~3s

## 🔧 MOTEUR DE SIMULATION

### État interne

```typescript
interface SimulationState {
  scenarioId: string           // scénario actif
  stepIdx: number              // index de l'étape en cours
  isPlaying: boolean           // play/pause
  particleProgress: number     // 0–1, progression de la particule
  visitedSteps: number[]       // étapes déjà visitées (pour le trail)
  pulsePhase: number           // 0–1 cyclique pour le glow actif
  speed: number                // 0.5×, 1×, 2×
  inputText: string            // texte input personnalisé (futur)
}
```

### Playback engine

Le moteur avance automatiquement les étapes quand `isPlaying` :

1. **Arrivée particule** : `particleProgress` animé de 0→1 via `requestAnimationFrame` + ease-out cubic (600ms)
2. **Dwell time** : Chaque step a un `duration` (ms). Le moteur attend cette durée avant d'avancer
3. **Avance** : `stepIdx++`, nouvelle particule lancée si l'organe change
4. **Fin** : `stepIdx >= steps.length` → `isPlaying = false`

### Contrôles

| Contrôle | Action |
|----------|--------|
| ▶ Play | Lance ou reprend la simulation |
| ‖ Pause | Suspend la simulation |
| ⏮ Reset | Remet à stepIdx=0, efface le trail |
| ◀ Step back | stepIdx-- (si > 0) |
| ▶ Step forward | stepIdx++ (si < steps.length) |
| Timeline click | Saute directement à l'étape cliquée |
| Scénario | Change le scénario et reset |
| Speed | 0.5× / 1× / 2× (futur) |

- **Tool calls** : badges individuels pour chaque appel MCP
- **Confidence** : barre de progression % (L3 uniquement)

La hauteur de la carte est calculée dynamiquement selon le contenu.

## ⚠️ CONTRAINTES

1. **Pas de nouvelle dépendance** : Tout doit être réalisé en SVG + React + CSS. Pas de Lottie, pas de GSAP, pas de Framer Motion.
2. **Performance** : Les animations doivent tourner à 60fps sur un écran 1080p. `requestAnimationFrame` uniquement, pas de `setInterval` pour les particules.
3. **Accessibilité** : Les contrôles doivent être navigables au clavier. Les labels doivent être lisibles.
4. **Responsive** : Le viewBox SVG fixe + `preserveAspectRatio="xMidYMid meet"` gère le scaling automatiquement.
5. **Pas de logique métier** : Le Signal est un **simulateur**, pas un moteur cognitif. Il ne fait jamais de routage ECS, d'Auto-Check ou de cristallisation — il les montre.

## 📐 TECHNICAL NOTES

### SVG `scale` vs `transform` (bug corrigé)

La propriété CSS individuelle `scale` ne fonctionne PAS sur les éléments SVG `<g>`. Solution : imbriquer deux `<g>` :
- Extérieur : `<g transform="translate(x,y)">` pour le positionnement SVG
- Intérieur : `<g style={{ transform: 'scale(N)', transformOrigin: '0 0' }}>` pour le scale CSS animable

Ce pattern est appliqué sur TOUS les composants nœud (NodeGfx, OrganNode, ApexNode, MutationNode, DriftNode, ClusterNode, MemoryNode, MCPToolNode, TimelineEventMarker).

### viewBox et centrage

Le viewBox fixe `0 0 1000 700` avec `preserveAspectRatio="xMidYMid meet"` suffit pour le responsive. Le centre du ring est à `(500, 370)`, avec 70px de marge au-dessus de Σ pour les annotations.
